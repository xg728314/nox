# STEP-027 — ACCOUNT RECOVERY DESIGN LOCK

## OBJECTIVE

현재 NOX는 signup → pending → approval → login 흐름은 동작하지만,
아이디 찾기 / 비밀번호 재설정의 셀프서비스 복구 흐름은 아직 존재하지 않는다.

이번 라운드의 목적은
**NOX 운영 구조에 맞는 계정 복구(account recovery) 정책을 설계로 고정(lock)하는 것**이다.

이 단계에서는 구현하지 않는다.
먼저 무엇을 운영자 처리로 둘지,
무엇을 사용자 셀프서비스로 열지,
보안상 어떤 정보까지 노출할지 문서로 확정한다.

---

## CURRENT CODE REALITY

감사 결과 기준 현재 상태:

- forgot-password UI 없음
- reset-password UI 없음
- 관련 API route 없음
- resetPasswordForEmail 호출 없음
- 기존 "비밀번호 찾기" 버튼은 dead UI였고 제거됨

즉 현재 실제 운영은:
- 아이디 분실 → 운영자 도움 필요
- 비밀번호 분실 → 운영자 재설정 필요
- 사용자 셀프서비스 복구는 아직 없음

---

## LOCKED BUSINESS CONTEXT

현재 회원가입 신청 시 수집하는 핵심 정보:

- 소속 매장
- 이름
- 닉네임
- 전화번호
- 이메일
- 비밀번호

로그인 ID는 현재 이메일 기반이다.
전화번호는 운영 식별용이다.

---

## DESIGN GOAL

복구 정책은 다음 둘 사이를 명확히 구분해야 한다:

1. 운영자 확인이 필요한 복구
2. 사용자 셀프서비스로 열 수 있는 복구

그리고 아래를 반드시 답해야 한다:

- 아이디(이메일) 분실 시 무엇을 기준으로 확인할 것인가
- 이메일을 사용자에게 얼마나 보여줄 것인가
- 비밀번호 재설정을 운영자 처리로 둘 것인가, 셀프서비스로 열 것인가
- 승인 대기 계정 / 거절 계정 / 정지 계정에 대해 복구 흐름이 어떻게 달라지는가

---

## REQUIRED DESIGN QUESTIONS

### 1. 아이디 찾기 정책

사용자가 아이디(이메일)를 잊었을 때,
어떤 입력값으로 본인 확인을 할지 설계할 것.

후보 기준:
- 소속 매장
- 이름
- 닉네임
- 전화번호

반드시 판단:
- 어떤 조합을 요구할지
- 결과로 이메일 전체를 보여줄지
- 일부 마스킹만 보여줄지
- 운영자가 확인 후 알려주는 방식으로 둘지

주의:
- 이메일 전체 노출은 보안 리스크가 있다
- 하지만 운영 효율도 고려해야 한다

---

### 2. 비밀번호 재설정 정책

반드시 판단:
- 운영자가 직접 재설정하는 방식이 기본인지
- Supabase email reset 기반 셀프서비스를 열 것인지
- 둘 다 지원할지

현재 코드베이스 현실을 기준으로 가장 안전한 1차 방식을 제안할 것.

중요:
- 지금은 reset flow 구현이 없으므로
- "당장 무엇이 가장 현실적이고 안전한가"를 기준으로 설계할 것

---

### 3. 승인 상태별 복구 처리

회원 상태에 따라 복구 정책이 달라지는지 정할 것.

대상:
- pending
- approved
- rejected
- suspended

예:
- pending 사용자는 아이디 확인만 가능?
- rejected/suspended 계정은 reset을 허용할지?
- approved만 셀프서비스 reset 허용할지?

---

### 4. 운영자 도구 필요 여부

현재 운영자가 계정 복구를 처리하려면
어떤 정보를 볼 수 있어야 하는지 설계할 것.

예:
- store
- full_name
- nickname
- phone
- email
- status

그리고 이 정보가
현재 approvals / admin / dashboard 어디에 있는지,
아니면 새 운영자 도구가 필요한지 판단할 것.

---

### 5. 사용자 UX 문구

login 화면 또는 recovery 화면에서
사용자에게 어떤 문구를 보여야 하는지 설계할 것.

예:
- 아이디를 잊으셨나요?
- 비밀번호를 잊으셨나요?
- 운영자에게 문의하세요
- 승인된 계정만 복구 가능합니다

주의:
- 아직 없는 기능을 이미 있는 것처럼 보이게 하면 안 됨

---

### 6. 1차 운영 정책과 2차 확장 정책 구분

반드시 나눠서 설계할 것:

#### 1차 (지금 바로 현실적인 정책)
예:
- 아이디 찾기 = 운영자 확인형
- 비밀번호 재설정 = 운영자 처리형

#### 2차 (나중에 확장 가능한 정책)
예:
- 이메일 마스킹 기반 아이디 찾기
- resetPasswordForEmail 기반 비밀번호 재설정

즉,
지금 바로 구현 가능한 것과
나중에 확장할 것을 분리해서 써야 한다.
